# Linux Ops Agent 答辩问题分析

> 本文档从评委/老师视角出发，系统梳理答辩中可能被提出的问题，并按类别分组。每个问题均附参考回答要点。

---

## 一、架构设计类（高频）

### Q1：为什么要用 Agent Skills 模式，而不是直接让 AI 生成 Shell 命令执行？

**考察点**：对 AI 安全边界的理解

**参考回答**：
- AI（DeepSeek）的输出具有不确定性，直接让 AI 生成并执行 Shell 命令存在严重安全隐患（如 `rm -rf`、`shutdown` 等）。
- Agent Skills 模式将"可执行的操作"限定为预先登记、人工审核过的 YAML 配置文件，AI 只负责**选择**而非**生成**命令。
- 这类似于 Function Calling 的安全最佳实践：让模型选工具，不让模型写工具。
- 同时 YAML skill 配置也记录了每个 skill 的风险等级（`risk_level`）和允许的命令清单（`allowed_commands`），形成双重审计链。

---

### Q2：你的系统架构中 Planner → Skill Loader → Executor → Report 这几层各自职责是什么？如果去掉 Planner 会怎样？

**考察点**：架构分层理解

**参考回答**：
| 层 | 职责 |
|---|---|
| Planner | 接收自然语言，判断意图，选择 skill（DeepSeek/关键词双模式），拒绝危险请求 |
| Skill Loader | 解析 `skills/*.yaml` 配置，校验脚本路径和必填字段，构建 Skill 对象 |
| Executor | 安全检查（路径合法性、风险等级），调用 `subprocess.run` 执行 Shell 脚本，设置超时，收集 stdout/stderr |
| Report | 汇总执行结果，调用 DeepSeek 生成 Markdown 诊断报告，或退化到本地模板 |

如果去掉 Planner，用户必须手动指定 `--skill` 参数，丧失自然语言交互能力。Planner 是系统"智能理解"的唯一入口。

---

### Q3：为什么选择 YAML 而不是 JSON 或数据库来存储 skill 配置？

**考察点**：技术选型依据

**参考回答**：
- YAML 可读性强，适合人工编写和审核运维 skill。
- 每个 YAML 文件 = 一个独立 skill，增删 skill 不影响其他文件（低耦合）。
- YAML 支持注释（JSON 不支持），方便在 skill 配置中说明每个命令的用途。
- 对于项目规模（5 个 skill），YAML 足够；如果未来扩展到上百个 skill，可考虑 SQLite 管理元数据。

---

### Q4：你的 Planner 有两种工作模式（DeepSeek 和关键词），什么情况下会退化到关键词模式？这样设计有什么好处？

**考察点**：容错设计和降级策略

**参考回答**：
退化触发条件（按优先级）：
1. `--no-llm` 命令行参数显式禁用。
2. `DEEPSEEK_API_KEY` 环境变量未设置（`.env` 不存在或为空）。
3. `openai` 依赖包未安装。
4. DeepSeek API 网络请求失败/超时。
5. DeepSeek 返回的 JSON 解析失败。

好处：
- 保证核心功能**始终可用**，不依赖外部 API。
- 答辩现场如果网络不稳定，可以立即切换到 `--no-llm` 模式演示。
- 符合"降级优雅"的工程原则（graceful degradation）。

---

### Q5：如果用户输入"帮我清理磁盘空间"，你的系统会怎么做？

**考察点**：安全边界和拒绝逻辑

**参考回答**：
系统会在 Planner 阶段被 `_looks_dangerous()` 方法拦截。"清空"关键词匹配到 `DANGEROUS_PATTERNS` 正则列表，返回 `refused=True`，并输出：

> 系统不会自动执行删除、格式化、关机、重启等破坏性操作。

不会进入 Executor 阶段。系统只做**诊断**，不做**变更操作**。这是项目的核心安全设计原则。

## 二、安全设计类（高频）

### Q6：你的安全机制具体有哪些？如果攻击者修改了 YAML 文件中的 script 路径怎么办？

**考察点**：纵深防御理解

**参考回答**：
多层安全机制：

1. **Planner 层**：正则匹配危险关键词（`rm -rf`、`shutdown`、`reboot`、`mkfs` 等），直接拒绝。
2. **Skill Loader 层**：校验脚本必须存在于 `scripts/` 目录且为普通文件。
3. **Executor 层**：`_validate()` 方法强制要求脚本路径以 `scripts/` 开头（`resolve()` 后比对），拒绝符号链接跳转到外部路径。同时检查 `risk_level`，`high` 级别默认禁用。
4. **执行层**：`subprocess.run` 使用 `capture_output=True`，脚本输出不会直接写入终端；设置 `timeout` 防止死循环；使用 `check=False` 防止异常中断。
5. **配置层**：`allow_high_risk` 默认为 `false`。

针对"修改 YAML 中的 script 路径"：Executor 的 `_validate()` 使用 `script.resolve()` 获取绝对路径后，与 `scripts/` 目录做前缀比对，即使 YAML 写 `../../etc/passwd` 也会被拦截。

---

### Q7：你的系统在普通用户权限下运行`df`、`ps`、`ss`这些命令安全吗？会不会被用来提权？

**考察点**：Linux 权限模型和安全边界

**参考回答**：
- `df`、`ps`、`ss`、`free`、`uptime`、`ping` 等都是 Linux 下**任何用户均可执行**的只读诊断命令，不会修改系统状态。
- 项目不会以 root 身份运行，也不在脚本中使用 `sudo`。
- 即使攻击者通过某种手段替换了 `scripts/` 下的脚本文件，Executor 的超时机制（默认 25 秒）也会限制恶意脚本的执行时间。
- 进一步加固方向：可以对脚本文件做 SHA256 校验。

---

### Q8：AI 模型本身的 prompt injection 攻击怎么防范？用户在输入中嵌入恶意指令怎么办？

**考察点**：LLM 安全

**参考回答**：
目前项目的防范措施：
1. Planner 在调用 DeepSeek 之前，先对用户原始输入做危险关键词正则检查（`_looks_dangerous`），拦截明显的恶意输入。
2. DeepSeek 的 system prompt 明确限制"禁止输出任意 Shell 命令"、"只能从给定 skills 中选择"。
3. 即使 DeepSeek 返回了非预期内容（如虚构一个 `skill_name`），Executor 也只执行 `scripts/` 目录下实际存在的脚本。
4. DeepSeek 返回的 JSON 中 `skills` 数组会与 `self.skills` 字典做交集过滤（`if name in self.skills`），虚构的 skill 被丢弃。

局限性（可坦诚说明）：
- 目前未对 DeepSeek 的 `answer` 字段做 XSS/注入过滤（但 Markdown 报告由本地生成，不存在 RCE 风险）。
- 后续可增加输入长度限制、敏感字符过滤等。

## 三、实现细节类

### Q9：`subprocess.run` 的参数 `capture_output=True`、`check=False`、`text=True`、`timeout` 各有什么作用？为什么选这些值？

**考察点**：Python subprocess 模块掌握程度

**参考回答**：
| 参数 | 作用 | 选择原因 |
|---|---|---|
| `capture_output=True` | 捕获 stdout/stderr 到内存 | 避免脚本输出直接污染终端，便于后续报告生成 |
| `check=False` | 命令失败不抛异常 | 诊断脚本即使 `exit_code != 0` 也可能有有用输出（如权限不足的提示） |
| `text=True` | 以字符串而非 bytes 返回输出 | 方便 Markdown 报告处理和 DeepSeek 总结 |
| `timeout=25` | 超时自动终止 | 防止诊断脚本因文件锁、NFS 挂载卡死等原因无限阻塞 |

---

### Q10：你的 Shell 脚本中用了哪些文本处理技术？举一个具体例子说明管道和 awk 的配合。

**考察点**：Shell 编程能力

**参考回答**（以 `disk_check.sh` 为例）：

```bash
df -P 2>/dev/null | awk 'NR > 1 {
  pct=$5
  gsub("%", "", pct)
  if (pct + 0 >= 80) {
    print $0
  }
}'
```

- `df -P` 以 POSIX 格式输出磁盘使用信息
- `2>/dev/null` 丢弃错误输出（如挂载点无权限）
- 管道 `|` 将 df 输出传给 awk
- `awk` 跳过表头行（`NR > 1`），提取第 5 列使用率，去掉 `%` 符号后转为数字比较，过滤使用率 ≥80% 的文件系统
- `+0` 技巧：将字符串强制转为数字

另外 `log_analyze.sh` 中还有典型的 grep → sed → sort → uniq → sort → head 管道链，用于统计 SSH 攻击来源 IP TOP 10。

---

### Q11：`log_analyze.sh` 中为什么要做日志文件路径检测（`/var/log/auth.log` → `/var/log/secure` → `journalctl`）？

**考察点**：Linux 系统兼容性

**参考回答**：
不同 Linux 发行版的认证日志位置不同：
- Debian/Ubuntu 系：`/var/log/auth.log`
- RHEL/CentOS/Fedora 系：`/var/log/secure`
- 使用 systemd 的系统：可通过 `journalctl -u sshd` 查询

脚本通过三级回退策略保证跨发行版兼容。还支持通过 `AUTH_LOG_FILE` 环境变量指定自定义路径（测试用）。

---

### Q12：DeepSeek API 调用中 `temperature=0.2` 的作用是什么？为什么不用默认值？

**考察点**：LLM API 参数理解

**参考回答**：
- `temperature` 控制输出的随机性，范围 0~2。
- 设为 0.2（偏低）是为了让 DeepSeek 在 skill 选择和报告总结时**输出更确定、更稳定**，减少随机选错 skill 或生成不准确诊断的可能性。
- 运维场景需要准确性和可复现性，不需要创意性。
- 默认值通常为 1.0，对于意图分类任务来说太高了。

---

### Q13：你的 `.env` 文件加载有两种方式（`python-dotenv` 和手动解析），为什么要手动实现一套？

**考察点**：容错设计

**参考回答**：
- `python-dotenv` 是外部依赖，可能未安装。
- 手动解析 `_load_dotenv_manually` 作为 fallback：逐行读取 `.env`，跳过注释和空行，按 `=` 分割，处理引号。
- 还做了环境变量去重：`if key not in os.environ`，已存在的环境变量不被 `.env` 覆盖。
- `_sanitize_proxy_env` 处理了 SOCKS 代理与 HTTP 代理冲突的兼容问题（有 HTTP 代理时移除 ALL_PROXY 的 socks5 设置）。

## 五、Linux 系统管理知识类

### Q16：`df -h` 和 `df -i` 的区别是什么？为什么运维中两个都要检查？

**考察点**：Linux 文件系统基础

**参考回答**：
- `df -h`：显示磁盘空间使用（以人类可读格式），关注的是**数据块**耗尽
- `df -i`：显示 inode 使用情况，关注的是**inode 节点**耗尽

两者都需要检查的原因：
- 磁盘空间满了 → 无法写入新数据
- inode 满了（即使磁盘还有空间）→ 无法创建新文件（每个文件需要一个 inode）
- 常见场景：大量小文件（如邮件队列、session 文件、缓存文件）耗尽 inode，但 `df -h` 显示空间充足

---

### Q17：`ss -tulnp` 和 `netstat -tulnp` 有什么区别？为什么选择 `ss`？

**考察点**：Linux 网络命令演进

**参考回答**：
- `netstat` 来自 net-tools 包，已逐渐被弃用
- `ss` 是 `netstat` 的现代替代品，来自 iproute2 包
- `ss` 直接从内核获取 socket 信息，速度更快（尤其在大量连接时）
- 参数含义：`-t` TCP、`-u` UDP、`-l` 监听中、`-n` 数字格式（不做 DNS 反向解析，加快输出）、`-p` 显示进程信息

---

### Q18：Linux 负载（load average）的三个值分别代表什么？高负载一定是 CPU 不够吗？

**考察点**：系统性能理解

**参考回答**：
`uptime` 输出的三个值：过去 1 分钟、5 分钟、15 分钟的平均负载（可运行 + 不可中断睡眠状态的进程数）。

高负载不一定是 CPU 不够：
- CPU-bound：计算密集型进程多
- I/O-bound：大量进程等待磁盘 I/O（`D` 状态）
- 进程数过多导致调度开销大

所以 `process_check.sh` 同时检查了 uptime、free（内存）、top（CPU）和 ps 进程状态（`R`/`D`/`S`），多维度诊断。

## 七、AI/LLM 相关类（新兴热点）

### Q21：为什么选择 DeepSeek 而不是 OpenAI GPT 或其他模型？

**考察点**：技术选型依据

**参考回答**：
- 课程项目要求使用国产大模型（或国内可访问的模型）
- DeepSeek 提供 OpenAI 兼容的 API（`/v1/chat/completions`），可以直接用 `openai` Python SDK 调用，迁移成本低
- DeepSeek 中文理解能力强，适合中文运维指令识别
- 成本低（DeepSeek-V4-Flash 免费或极低价格）

代码中也体现了灵活性：通过 `config.yaml` 和 `.env` 可随时切换 base_url 和 model，换成 Qwen、GLM 等也只需改配置。

---

### Q22：DeepSeek 返回的不是合法 JSON 怎么办？你的 `_parse_json_object` 做了哪些容错处理？

**考察点**：LLM 输出的鲁棒处理

**参考回答**：
`_parse_json_object` 的三层容错：
1. 去除 markdown 代码块标记（```` ```json ```` 和 ``` ``` ```）
2. 用 `text.find("{")` / `text.rfind("}")` 提取 JSON 对象的起止位置（容忍 LLM 在前后附加解释文字）
3. 如果找不到 `{` 和 `}`，抛出 `ValueError` 被上层 `_plan_with_llm` 捕获，返回 `None` 触发回退到关键词模式

这是一种实用的"宽松解析"策略，比直接 `json.loads(text)` 更鲁棒。

---

### Q23：你把 Shell 脚本的输出（最多 12000 字符）直接喂给 LLM，有没有考虑过上下文长度限制？

**考察点**：LLM 工程实践

**参考回答**：
- DeepSeek-V4 的上下文窗口足以处理 12000 字符（约 3000-4000 tokens）。
- `summarize_report` 中对 `raw_output` 做了 `[:12000]` 截断保护。
- 如果未来 skill 输出更大，可以增加预处理步骤（如只提取关键指标、错误行、异常值），减少 token 消耗。
- 当前项目规模下这不是瓶颈。

### Q25：`doctor` 命令检查了哪些内容？为什么答辩要设计这个命令？

**考察点**：工程实用性

**参考回答**：
`doctor` 检查 7 类共 15+ 项：
1. Python 版本（≥3.10）
2. Python 依赖包（openai, dotenv, yaml, pytest）

答辩意义：30 秒内完成环境验证，向评委证明项目是"真实可运行"的，不是 PPT 项目。

## 九、可能的追问/深挖问题

### Q26：如果 DeepSeek 选择了不合适的 skill（比如用户说"网络慢"却选了 disk_check），怎么办？

**参考回答**：
- 这种情况在 LLM 应用中确实存在（幻觉/误判）。
- 当前缓解措施：YAML 中的关键词列表提供第二层匹配，关键词模式的确定性结果可作为参照。
- 改进方向：在 DeepSeek prompt 中增加 few-shot 示例；增加用户确认环节（"将执行 disk_check，是否继续？"）。

---

### Q27：多个 skill 的执行是串行的，如果改为并行执行会有什么问题？

**参考回答**：
- 当前设计是有意串行的：每个 skill 的输出打印到终端，串行执行方便用户实时观察进度。
- 并行化的挑战：
  - 终端输出会交错，可读性差
  - 如果 health_report 依赖其他 skill 的输出，并行会破坏依赖关系
  - 某些诊断命令（如 `ping`）有网络副作用，并行可能相互干扰
- 可以用 `concurrent.futures` 改造，但当前规模下收益不大。

---

### Q28：配置文件中的 `max_skills_per_request: 3` 为什么是 3 而不是更大？

**参考回答**：
- 防止一次请求触发过多诊断，导致执行时间过长（每个 skill 最多 25 秒）。
- DeepSeek 一次返回过多 skill 也容易出错。
- 3 是一个经验值，覆盖了常见的"磁盘+日志"或"进程+网络"组合。
- 用户如需全部巡检，可以使用 `daily` 命令触发 `health_report`（一个 skill 涵盖所有检查）。
