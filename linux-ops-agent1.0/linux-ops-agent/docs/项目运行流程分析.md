# 项目运行流程全景分析

> 本文档从代码层面逐层追踪一次完整请求从输入到报告生成的全过程。

---

## 一、总览：一条请求的 6 个阶段

```
用户输入 "检查磁盘空间问题"
        │
   ┌────▼─────┐
   │ 1. 入口层  │  ops 脚本 / main.py CLI
   └────┬─────┘
        │
   ┌────▼─────┐
   │ 2. 配置层  │  load_config() — 读取 config.yaml
   └────┬─────┘
        │
   ┌────▼─────┐
   │ 3. 加载层  │  load_skills() — 解析 skills/*.yaml
   └────┬─────┘
        │
   ┌────▼─────┐
   │ 4. 规划层  │  Planner.plan() — DeepSeek / 关键词 / 安全拦截
   └────┬─────┘
        │
   ┌────▼─────┐
   │ 5. 执行层  │  SkillExecutor.run() — subprocess 执行 Shell
   └────┬─────┘
        │
   ┌────▼─────┐
   │ 6. 报告层  │  create_report() — DeepSeek 总结 + Markdown 输出
   └──────────┘
```

## 二、阶段 1：入口层 — 两种启动方式

### 方式 A：`./ops` 便捷脚本（`ops` 文件）

```bash
# ops 脚本核心逻辑（伪代码）
ROOT_DIR = 脚本所在目录

# 优先用虚拟环境中的 Python
if .venv/bin/python 存在且可执行:
    PYTHON = .venv/bin/python
else:
    PYTHON = python3

# 无参数 → 进入交互模式
if 参数数量 == 0:
    exec python main.py --interactive

# 单参数 --no-llm → 无 LLM 的交互模式
if 参数数量 == 1 and 参数 == "--no-llm":
    exec python main.py --interactive --no-llm

# 有参数 → 单次执行模式，参数原样传递
else:
    exec python main.py "检查磁盘空间问题"
```

设计意图：
- 用户不需要每次输入 `python main.py`，也不需要给中文加引号
- 自动检测虚拟环境，避免环境问题
- `exec` 保证进程替换，退出码直接传递

### 方式 B：`python main.py` 直接调用

```python
# main.py 入口
def main(argv=None):
    args = build_parser().parse_args(argv)
    if args.interactive:
        return interactive_main(use_llm=not args.no_llm)

    config = load_config(PROJECT_ROOT)      # → 阶段 2
    skills = load_skills(PROJECT_ROOT)      # → 阶段 3
    # ... 继续阶段 4-6
```

## 三、阶段 2：配置层 — `agent/config.py` 的 `load_config()`

### 调用链

```
load_config(PROJECT_ROOT)
  ├── _read_yaml("config.yaml")           # 读取并解析 YAML
  │     ├── yaml.safe_load(file_content)  # 安全加载（禁止任意 Python 对象）
  │     ├── 校验：文件必须存在
  │     └── 校验：内容必须是 dict
  │
  └── 构造三层配置对象（dataclass frozen）
        ├── DeepSeekConfig(base_url, model, timeout_seconds)
        ├── ExecutionConfig(script_timeout_seconds, reports_dir, allow_high_risk)
        └── AgentConfig(fallback_to_keywords, max_skills_per_request)
```

### config.yaml 默认值

```yaml
deepseek:
  base_url: https://api.deepseek.com
  model: deepseek-v4-flash
  timeout_seconds: 30

execution:
  script_timeout_seconds: 25      # 每个脚本最多执行 25 秒
  reports_dir: reports            # 报告输出目录（相对路径 → 转为绝对路径）
  allow_high_risk: false          # 默认禁止高风险 skill

agent:
  fallback_to_keywords: true      # DeepSeek 失败回退关键词
  max_skills_per_request: 3       # 单次最多选择 3 个 skill
```

### 配置对象特点

- 所有配置类使用 `@dataclass(frozen=True)` **不可变**，防止运行时意外修改
- `reports_dir` 从相对路径转为 `project_root / "reports"` 绝对路径
- 每个字段有 `get(key, default)` 防御性取值

## 四、阶段 3：加载层 — `agent/skill_loader.py` 的 `load_skills()`

### 调用链

```
load_skills(PROJECT_ROOT)
  │
  ├── 检查 skills/ 目录是否存在
  │
  └── 遍历 skills/*.yaml，逐个调用 _load_skill_file()
        │
        ├── yaml.safe_load(file_content)
        │
        ├── 校验必填字段：name, description, script, risk_level, allowed_commands
        │     ├── 缺少任一字段 → ValueError
        │     └── allowed_commands 为空 → ValueError
        │
        ├── 构造 SkillInput 列表（inputs 字段）
        │     └── 每个 input: {name, type, default, required}
        │
        ├── 构造 Skill 对象（dataclass frozen）
        │     ├── name: str
        │     ├── description: str
        │     ├── script: Path        ← project_root / "scripts/disk_check.sh"
        │     ├── risk_level: str     ← "low" / "medium" / "high"
        │     ├── allowed_commands: tuple[str, ...]
        │     ├── keywords: tuple[str, ...]
        │     └── inputs: tuple[SkillInput, ...]
        │
        ├── 校验：skill name 不可重复
        ├── 校验：脚本文件必须存在
        └── 校验：脚本必须是普通文件（非目录/符号链接指向目录）
```

## 五、阶段 4：规划层 — `agent/planner.py` 的 `Planner.plan()`

这是系统最核心的决策环节。根据用户请求确定执行哪些 skill（或拒绝）。

### plan() 主流程

```
plan(user_request, use_llm=True)
  │
  ├── 第 0 步：安全检查 _looks_dangerous()
  │     ├── 将用户输入转小写
  │     ├── 用正则列表 DANGEROUS_PATTERNS 逐一匹配
  │     │     ├── \brm\s+-rf\b         # rm -rf
  │     │     ├── 删除所有
  │     │     ├── 清空
  │     │     ├── 格式化
  │     │     ├── \bmkfs\b             # mkfs 格式化
  │     │     ├── \bshutdown\b
  │     │     ├── \breboot\b
  │     │     ├── 重启服务器
  │     │     └── 关闭服务器
  │     ├── 命中 → 返回 Plan(refused=True)    ← 终止！
  │     └── 未命中 → 继续
  │
  ├── 第 1 步：尝试 DeepSeek 规划（如果 use_llm=True 且 API 可用）
  │     └── _plan_with_llm(user_request)
  │           ├── 构造 skill 目录 JSON
  │           ├── 拼装 system prompt（见下方）
  │           ├── DeepSeekClient.complete(prompt) → 返回 JSON 字符串
  │           ├── _parse_json_object(raw)         # 宽松 JSON 解析
  │           ├── 提取 skills 数组 + answer 字段
  │           ├── 过滤：只保留在 self.skills 中真实存在的 skill → 防幻觉
  │           ├── 截断：只取前 max_skills_per_request 个
  │           ├── 有 skill → Plan(skills=..., source="deepseek")
  │           ├── 无 skill → Plan(skills=(), answer="...", source="deepseek")
  │           └── 异常 → 返回 None（触发降级）
  │
  └── 第 2 步：关键词降级（LLM 未用或失败时）
        └── _plan_with_keywords(user_request)
              ├── 将用户输入转小写
              ├── 遍历所有 skill，计算匹配分数
              │     ├── keyword 命中 → +2 分
              │     └── skill name 中的 token 命中 → +1 分
              ├── 按分数降序排列，取前 N 个
              ├── 有命中 → Plan(skills=..., source="keyword")
              └── 无命中 → Plan(skills=(), reason="未命中明确关键词...")
```

### DeepSeek System Prompt 设计要点

```
你是 Linux 运维 Agent 路由器。
- 只能从给定 skills 中选择，禁止输出任意 Shell 命令
- 只有明确运维需求时才选 skill
- 闲聊/普通问题 → skills 为空，answer 字段正常回答
- 不要为了有结果而默认选择 health_report
- 必须返回严格 JSON 格式
```

这个 prompt 的设计意图：
1. **约束输出范围**：不能自由发挥
2. **防止过度匹配**：闲聊不触发诊断
3. **防止惰性匹配**：不确定时禁止默认选 health_report

### JSON 解析的三层容错（`_parse_json_object`）

```python
def _parse_json_object(text):
    text = text.strip()
    # 容错 1：去除 markdown 代码块标记
    if text.startswith("```"):
        text = re.sub(r"^```(?:json)?", "", text)   # 去掉开头的 ```json
        text = re.sub(r"```$", "", text)             # 去掉结尾的 ```
    # 容错 2：提取第一个 { 到最后一个 } 之间的内容
    start = text.find("{")
    end = text.rfind("}")
    if start == -1 or end == -1 or end < start:
        raise ValueError(...)
    # 容错 3：只解析 JSON 对象部分
    return json.loads(text[start:end+1])
```

这能处理以下 LLM 异常输出：
- ````json\n{"skills":...}\n````
- `好的，我选择：\n{"skills":["disk_check"]}\n以上是分析。`
- `{"skills":["disk_check"],"reason":"..."}`

## 六、阶段 5：执行层 — `agent/executor.py` 的 `SkillExecutor.run()`

### 执行前安全检查 `_validate()`

```
_validate(skill)
  │
  ├── 检查 1：脚本路径必须在 scripts/ 目录下
  │     └── script.resolve() 的绝对路径必须以 scripts_root + "/" 开头
  │     └── 不通过 → UnsafeSkillError (防止 "../../etc/passwd" 类攻击)
  │
  └── 检查 2：高风险 skill 是否被允许
        └── skill.risk_level == "high" and config.allow_high_risk == False
        └── 不通过 → UnsafeSkillError
```

### 执行过程

```python
def run(self, skill: Skill) -> ExecutionResult:
    # 1. 安全校验
    self._validate(skill)
    
    # 2. 记录开始时间
    started = time.monotonic()
    
    # 3. 执行 Shell 脚本
    try:
        completed = subprocess.run(
            ["bash", str(skill.script)],   # 以 bash 解释器执行
            cwd=self.project_root,          # 工作目录 = 项目根
            text=True,                      # 文本模式（str 非 bytes）
            capture_output=True,            # 捕获 stdout/stderr
            timeout=config.script_timeout_seconds,  # 25 秒超时
            check=False,                    # 失败不抛异常
        )
        return ExecutionResult(
            skill_name=skill.name,
            script=skill.script,
            stdout=completed.stdout,
            stderr=completed.stderr,
            exit_code=completed.returncode,
            duration_seconds=time.monotonic() - started,
            timed_out=False,
        )
    
    # 4. 超时处理
    except subprocess.TimeoutExpired as exc:
        return ExecutionResult(
            skill_name=skill.name,
            script=skill.script,
            stdout=exc.stdout or "",
            stderr=(exc.stderr or "") + f"\nTimed out after {timeout}s",
            exit_code=124,                   # 标准超时退出码
            duration_seconds=time.monotonic() - started,
            timed_out=True,
        )
```

### ExecutionResult 关键属性

```python
@dataclass(frozen=True)
class ExecutionResult:
    skill_name: str
    script: Path
    stdout: str
    stderr: str
    exit_code: int
    duration_seconds: float
    timed_out: bool = False
    
    @property
    def ok(self) -> bool:
        return self.exit_code == 0 and not self.timed_out
```

## 七、阶段 6：报告层 — `agent/report.py` 的 `create_report()`

### 报告生成流程

```
create_report(user_request, plan, selected_skills, results, reports_dir, llm_client, use_llm)
  │
  ├── 1. 确保 reports/ 目录存在
  │
  ├── 2. 生成文件名：20250608-143025-disk_check-log_analyze.md
  │     └── 格式：{timestamp}-{skill_names}.md
  │     └── skill 部分截断到 80 字符
  │
  ├── 3. 构建原始输出
  │     └── build_raw_output(results)
  │           └── 拼接每个 result 的：skill 名、脚本路径、退出码、耗时、stdout、stderr
  │
  ├── 4. 生成报告正文（两种策略）
  │     │
  │     ├── 策略 A：DeepSeek 总结（use_llm=True 且 API 可用 且 有输出）
  │     │     └── llm_client.summarize_report(user_request, raw_output)
  │     │           ├── System: "你是 Linux 运维诊断助手，写出简洁、可答辩展示的中文报告"
  │     │           ├── 要求六要素：问题概述、关键证据、可能原因、建议处理步骤、
  │     │           │            涉及的 Linux 命令、风险提示
  │     │           ├── raw_output 截断到 12000 字符
  │     │           └── 如果异常 → 降级到策略 B
  │     │
  │     └── 策略 B：本地模板（fallback_summary）
  │           └── 固定模板包含六要素框架
  │                 ├── 问题概述（填入 user_request）
  │                 ├── 关键证据（填入 raw_output 的代码块）
  │                 ├── 可能原因（通用引导文字）
  │                 ├── 建议处理步骤（1→2→3 通用步骤）
  │                 ├── 涉及的 Linux 命令（说明来自受控脚本）
  │                 └── 风险提示（只诊断不修改）
  │
  ├── 5. 写入文件头（HTML 注释元信息）
  │     └── <!-- generated by linux-ops-agent at 20250608-143025 -->
  │     └── <!-- selected skills: disk_check, log_analyze -->
  │
  └── 6. 写入文件并返回路径
```

## 八、完整追踪：一次"检查磁盘空间问题"的运行时数据流

### 阶段 4：Planner 决策

```
输入: "检查磁盘空间问题"

── DeepSeek 路径 ──
发送 prompt →
  "你是 Linux 运维 Agent 路由器..."
  "可用 skills: [{
     name: disk_check,
     description: 检查磁盘空间、inode 使用率、日志目录占用和大文件,
     keywords: [disk, df, du, inode, 磁盘, 空间, 容量, 大文件, 日志满]
   }, ...]"

  '{"skills":["disk_check"],"reason":"用户想检查磁盘空间使用情况","answer":""}'

### 阶段 5：Executor 执行

```
SkillExecutor.run(skills["disk_check"])
  │
  ├── _validate() ✓ (risk_level="low", 路径在 scripts/ 下)
  │
  └── subprocess.run(["bash", "scripts/disk_check.sh"], ...)
        │
        └── disk_check.sh 执行过程：
              ├── df -h          → 输出磁盘空间（人类可读）

              ├── find /var/log -size +100M → 查找 >100MB 的日志文件
              └── echo hints     → 诊断提示
              
              返回 stdout（假设）：
              Filesystem  Size  Used  Avail  Use%  Mounted on
              /dev/sda1    50G   42G   5.1G   90%  /
              ...
              
              exit_code = 0

返回 ExecutionResult(
    skill_name="disk_check",
    stdout="== Disk usage ==\nFilesystem  ...",
    exit_code=0,
    duration_seconds=0.87,
    timed_out=False,
)
```

### 特殊命令的识别逻辑

```
interactive_main 中的文本匹配（按顺序）：
  text in {"exit","quit","退出","q"}      → 退出
  text in {"help","帮助","?"}              → 打印帮助
  text in {"list","skills","技能"}         → 列出 skill
  text in {"reports","ls reports","报告列表"} → 报告列表
  text in {"last report","last","latest report","最新报告","查看最新报告"} → 最新报告
  text in {"doctor","check env","自检","环境检查"} → 环境自检
  text in {"clear reports","clean reports","清空报告","清除报告"} → 清理报告
  text in {"daily","report","巡检","报告"}   → 综合巡检
  其他 → 作为自然语言查询传给 main()
```

## 十、DeepSeek 客户端初始化流程

```
DeepSeekClient.__init__(config)
  │
  ├── _load_dotenv_if_available()
  │     └── 尝试 import dotenv → load_dotenv()
  │     └── ImportError → 静默跳过
  │
  ├── _load_dotenv_manually(".env")
  │     └── 读取 .env 文件，逐行解析
  │     └── 跳过注释(#)、空行、无=的行
  │     └── 去除引号
  │     └── 只在 key 不在 os.environ 时才设置（已有环境变量优先）
  │
  ├── _sanitize_proxy_env()
  │     └── 如果同时有 HTTPS_PROXY 和 ALL_PROXY=socks5://
  │     └── → 移除 ALL_PROXY（避免 openai SDK 用 SOCKS 代理导致连接失败）
  │
  └── 保存配置
        ├── api_key = os.getenv("DEEPSEEK_API_KEY")
        ├── model   = os.getenv("DEEPSEEK_MODEL", config.model)
        ├── base_url = os.getenv("DEEPSEEK_BASE_URL", config.base_url)
        └── _client = None  （延迟初始化）
```

### LLM 调用时机

```
DeepSeekClient.complete(prompt, system=None)
  │
  ├── 检查 api_key 是否存在
  │     └── 不存在 → raise RuntimeError + 设置 last_error
  │
  ├── _ensure_client()  # 延迟初始化
  │     ├── import openai（失败 → 诊断提示）
  │     └── OpenAI(api_key=..., base_url=...)
  │
  └── client.chat.completions.create(
        model=model,
        messages=[
          {role: "system", content: system},   # 可选
          {role: "user",   content: prompt},
        ],
        temperature=0.2,    # 低温度 → 更稳定的输出
        timeout=30,         # API 调用超时
      )
      → response.choices[0].message.content
```

## 十二、关键设计决策总结

| 决策 | 选择 | 理由 |
|---|---|---|
| 命令执行模式 | 预登记 Shell 脚本 | 安全可控，AI 不能随意生成命令 |
| 配置格式 | YAML | 可读、可注释、低耦合、Git 友好 |
| 配置对象 | frozen dataclass | 不可变，防止运行时篡改 |
| LLM 调用方式 | OpenAI 兼容 SDK | 可随时切换模型，不改代码 |
| 降级策略 | 三重回退（LLM→关键词→空） | 核心功能不依赖外部 API |
| 报告格式 | Markdown | 可读、可展示、可 Git 追踪 |
| JSON 解析 | 宽松提取（find { → rfind }） | 容忍 LLM 格式不规范 |
| 超时设置 | 脚本 25s + API 30s | 防止资源泄漏和无限等待 |
| 温度参数 | 0.2 | 运维场景需要确定性输出 |
| 安全级别 | 3 层（prompt→正则→路径校验） | 纵深防御，单点失效不影响整体 |
